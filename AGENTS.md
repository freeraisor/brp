# Инструкции для Codex

## Общий подход

- Работать по одной фиче за раз и держать изменения небольшими. Если не сказано что можно брать больше.
- Как можно меньше додумывать реализацию за пользователя, если он явно не дал на это разрешение.
- Если требований не хватает, сначала уточнять детали и предлагать несколько разумных вариантов решения.
- До изменения поведения разбираться в текущей структуре проекта и существующих паттернах.
- Отложенные задачи, которые не входят в текущую фичу, фиксировать в `BACKLOG.md`, чтобы они не терялись между итерациями.
- Не тратить время и токены на переводы на другие языки. Игнорировать поддержку файлов локализации.

## Код и поддерживаемость

- Соблюдать существующий код-стайл проекта.
- Писать простой, читаемый и поддерживаемый код.
- Не добавлять сложные абстракции без практической необходимости.
- Предпочитать локальные, понятные изменения вместо широких переделок.

## UI и Foundry

- Перед любой работой над UI (шаблоны, диалоги, CSS, рендер) сверяться с [FOUNDRY.md](./FOUNDRY.md) — там собраны нюансы рендеринга Foundry/BRP, которые уже проверены на граблях.
- Если в процессе задачи нашлась новая грабля по рендерингу Foundry — дополнять [FOUNDRY.md](./FOUNDRY.md) коротким пунктом, чтобы знание не терялось между итерациями.

## Рефакторинг

- Рефакторить старый код разрешено, если это помогает задаче, снижает сложность или улучшает поддерживаемость.
- Любой нетривиальный рефакторинг нужно кратко обосновывать перед внесением изменений.
- Не смешивать крупный рефакторинг с новой фичей без явной причины.

## Коммуникация

- До реализации неоднозначных решений описывать предположения и спрашивать подтверждение.
- Если есть несколько путей, кратко объяснять плюсы и минусы каждого.
- После изменений сообщать, какие файлы затронуты и как проверялась работа.
